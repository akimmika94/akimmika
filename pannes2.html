Méthodologie de dépannage
7 étapes à suivre systématiquement pour tout incident
Méthode
1
Identifier le problème
Qui est impacté ? (1 utilisateur ou tout le réseau) — Depuis quand ? — Quel symptôme précis ?
Toujours reformuler le problème clairement avant d'agir.
2
Reproduire le problème
Essayer de voir l'erreur soi-même. Tester sur une autre machine pour isoler si c'est lié au poste ou à l'infrastructure.
3
Isoler la cause — Approche OSI
Physique → câble branché ? — Réseau → IP correcte ? — Transport → ports ouverts ? — Application → service actif ?
4
Tester des hypothèses
Utiliser les outils de diagnostic adaptés au domaine (réseau, système, DNS...). Une hypothèse à la fois.
5
Corriger le problème
Redémarrer un service — Corriger une config — Libérer des ressources — Remplacer un matériel.
6
Vérifier
Le problème est-il résolu ? Tests utilisateurs OK ? Ne pas fermer le ticket tant que la résolution n'est pas confirmée.
7
Documenter
Cause — Solution — Temps de résolution. La documentation protège contre la récurrence et aide les collègues.

Pannes réseau
Connectivité, IP, DNS, accès Internet et ressources internes
Réseau
CAS 01
Plus d'accès au réseau
Un utilisateur ne peut accéder ni aux fichiers partagés ni à Internet.
Test
ipconfig
ping 192.168.1.1
Analyse : Ping passerelle en échec → problème réseau local (switch, câble, VLAN).
Solution : Vérifier câble → tester autre port switch → vérifier configuration VLAN.
CAS 02
Pas d'accès Internet — IP APIPA
Un utilisateur dit : "Je n'ai plus Internet sur mon poste."
Symptôme clé
IP = 169.254.x.x → adresse APIPA = pas de DHCP
Analyse : Le poste ne reçoit pas d'adresse IP du serveur DHCP.
Solution : Vérifier serveur DHCP, puis :
ipconfig /release → ipconfig /renew
Vérification
Nouvelle IP correcte (192.168.x.x) → ping 8.8.8.8 OK → navigation OK
CAS 03
Site web inaccessible — DNS KO
"Je ne peux pas accéder à google.com"
Tests
ping google.com → Échec
ping 8.8.8.8 → OK
Analyse : Réseau OK mais DNS KO. Le nom n'est pas résolu, l'IP est joignable.
Solution :
nslookup google.com → vérifier le DNS
Configurer DNS : 8.8.8.8
CAS 04
Partages réseau KO mais Internet OK
L'utilisateur peut aller sur Internet mais pas accéder à un serveur interne.
Tests
ping serveur → Échec
ping 8.8.8.8 → OK
Analyse : Problème réseau interne ou DNS interne défaillant.
Solution :
nslookup serveur → vérifier DNS interne
Ajouter DNS entreprise → vérifier firewall
CAS 05
Sites HTTPS inaccessibles
Certains sites ne s'ouvrent pas (HTTPS uniquement).
Test
curl https://site.com → Erreur certificat
Analyse : Problème de certificat ou date/heure système incorrecte.
Solution : Vérifier date/heure (date) → Synchroniser NTP → Vérifier certificats
CAS 06
Connexion très lente
Tout fonctionne mais très lentement.
Test
ping serveur → Latence élevée (ex : 300 ms)
Analyse : Congestion réseau ou matériel défectueux (câble, switch).
Solution : Vérifier switch → vérifier câble défectueux → vérifier utilisation réseau
CAS 07
SSH impossible — Port fermé
"Je n'arrive pas à me connecter en SSH"
Tests
ping serveur → OK
ssh user@serveur → Échec
telnet serveur 22 → Échec
Analyse : Port 22 fermé ou service SSH arrêté.
Solution :
systemctl status ssh
systemctl start ssh
CAS 08
Mauvaise passerelle configurée
L'utilisateur accède au réseau local mais pas à Internet.
Test
ipconfig → Passerelle incorrecte
Analyse : Mauvaise configuration réseau (IP fixe mal saisie ou DHCP défaillant).
Solution : Corriger passerelle (ex : 192.168.1.1) → vérifier DHCP
CAS 09
DNS lent ou défaillant
Les sites mettent du temps à charger.
Test
nslookup google.com → Réponse très lente
Analyse : Serveur DNS surchargé ou hors service.
Solution : Changer DNS (ex : 8.8.8.8) → vérifier DNS interne
🔀
VLAN & Routage
Isolation, trunk, inter-VLAN, NAT, DHCP relay
VLAN / Routage
CAS 10
VLAN mal configuré — Machines isolées
Un utilisateur ne voit pas certains équipements. Ping certains postes OK, d'autres KO.
Analyse : Mauvais VLAN sur un port switch.
Solution :
show vlan brief → vérifier port
switchport access vlan 20
CAS 11
VLAN non propagé sur trunk
Un VLAN fonctionne sur un switch mais pas sur un autre.
Analyse : VLAN non autorisé sur le trunk entre switches.
Solution :
switchport mode trunk
switchport trunk allowed vlan 10,20
CAS 12
Pas de communication inter-VLAN
VLAN 10 et VLAN 20 ne communiquent pas.
Analyse : Pas de routage inter-VLAN configuré.
Solution :
Configurer routage via routeur (router-on-a-stick)
OU via switch L3 + ip routing
CAS 13
Boucle réseau (loop)
Réseau très lent ou instable. Broadcast élevé, switch saturé.
Analyse : Câble branché en double → boucle réseau.
Solution : Débrancher câble en trop → activer STP :
spanning-tree portfast
CAS 14
DHCP ne fonctionne pas dans un VLAN
Les postes d'un VLAN n'ont pas d'IP (169.254.x.x).
Analyse : DHCP non relayé entre le VLAN et le serveur DHCP.
Solution : Configurer relais DHCP sur l'interface du routeur/switch L3 :
ip helper-address 192.168.1.10
CAS 15
NAT mal configuré
Les machines internes ne vont pas sur Internet.
Test
ping 8.8.8.8 → Échec depuis les postes internes
Analyse : NAT absent ou incorrect sur le routeur.
Solution : Configurer NAT/PAT sur le routeur :
ip nat inside (côté LAN)
ip nat outside (côté WAN)
CAS 16
Problème de routage — Route absente
Impossible d'accéder à un autre réseau.
Test
tracert serveur → Blocage sur un routeur
Analyse : Route absente ou incorrecte dans la table de routage.
Solution :
show ip route → vérifier
Ajouter route statique manquante
CAS 17
Routage asymétrique
Ping fonctionne dans un sens mais pas dans l'autre.
Analyse : PC → serveur OK, serveur → PC KO = route de retour absente.
Solution : Ajouter route retour → vérifier tables de routage des deux côtés
CAS 18
VLAN natif mal configuré
Problèmes aléatoires entre switches.
Analyse : VLAN natif différent des deux côtés du trunk.
Solution : Uniformiser le VLAN natif sur tous les trunks :
switchport trunk native vlan 1
🖥️
Pannes système & serveur
CPU, disque, services, comptes, ports
Système
CAS 19
Serveur web inaccessible
"Le site de l'entreprise ne répond plus"
Tests
ping serveur → OK
curl http://serveur → Échec
Analyse : Réseau OK → problème applicatif. Le service web est arrêté.
Solution :
systemctl status apache2
systemctl start apache2
CAS 20
Poste très lent — CPU 100%
"Mon ordinateur est super lent"
Test
top → CPU à 100%, processus suspect
Analyse : Processus gourmand en ressources.
Solution : Identifier le processus → kill -9 PID
CAS 21
Plus d'espace disque
"Je ne peux plus enregistrer de fichiers"
Test
df -h → Disque à 100%
Analyse : Saturation disque (logs, fichiers temporaires).
Solution :
Supprimer fichiers inutiles
rm -rf /var/log/*.log
CAS 22
Service ne démarre plus après reboot
Après redémarrage, une application ne démarre plus automatiquement.
Test
systemctl status nom_du_service → Failed
Analyse : Service non activé au démarrage ou erreur de configuration.
Solution :
systemctl enable nom_du_service
systemctl start nom_du_service
CAS 23
Utilisateur ne peut plus se connecter
Login refusé sur un poste ou serveur.
Analyse : Mot de passe incorrect, compte verrouillé ou problème Active Directory.
Solution : Déverrouiller compte → réinitialiser mot de passe → vérifier droits AD
CAS 24
Erreur "port déjà utilisé"
Un service ne démarre pas — conflit de port.
Test
netstat -tulnp → Port déjà utilisé par un autre processus
Analyse : Deux services tentent d'utiliser le même port.
Solution : Identifier le processus conflictuel → l'arrêter ou changer le port
CAS 25
Sauvegarde échoue
Backup automatique en erreur.
Analyse : Disque plein / accès réseau perdu / droits insuffisants.
Solution : Vérifier logs → libérer espace → corriger permissions
🖨️
Pannes d'impression
Spooler, file d'attente, réseau imprimante, droits
Impression
CAS 26
Impossible d'imprimer — Spooler bloqué
Un utilisateur envoie un document, rien ne sort. Document en "Erreur" dans la file.
Analyse : Spooler bloqué ou file d'attente saturée.
Solution :
net stop spooler
net start spooler
CAS 27
Imprimante hors ligne
L'imprimante apparaît "hors ligne" sur le PC.
Test
ping IP_imprimante → Échec
Analyse : Problème réseau ou imprimante éteinte. IP souvent changée si DHCP.
Solution : Vérifier alimentation → câble réseau/Wi-Fi → vérifier IP (souvent change avec DHCP → préférer IP fixe)
CAS 28
Documents bloqués en file
Les documents s'accumulent mais ne sortent pas.
Analyse : Service spooler ou driver défectueux.
Solution : Vider la file → redémarrer spooler → réinstaller pilote si besoin
CAS 29
Seuls certains utilisateurs ne peuvent pas imprimer
Un groupe d'utilisateurs est bloqué, d'autres impriment normalement.
Analyse : Problème de droits sur le serveur d'impression.
Solution : Vérifier permissions sur serveur d'impression → ajouter utilisateurs au bon groupe

⚠️
Erreurs fréquentes à éviter
Les pièges classiques du dépannage
Attention
⚠️ Ne jamais faire
Aller trop vite — changer quelque chose sans comprendre le problème
Ne pas tester après chaque modification
Ignorer les logs — ils contiennent souvent la réponse directe
Multiplier les changements en même temps — impossible d'identifier la cause
Ne pas communiquer avec l'utilisateur pendant l'intervention
Fermer un ticket sans vérification de la résolution côté utilisateur
Oublier de documenter — la même panne reviendra sans trace de la solution
✅ Toujours
🔍 Observer avant d'agir  →  🧪 Tester méthodiquement  →  🧠 Comprendre la cause  →  🔧 Corriger  →  ✔️ Vérifier

Et surtout : Ne pas paniquer. Raisonner avec le modèle OSI. Lire les messages d'erreur.
