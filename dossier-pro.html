02 CCP1 — Ex.2 : Partages, droits d'accès & permissions AGDLP
Contexte & environnement
Type TP individuel — salle de formation
Période 12/11/2025 → 13/11/2025
Domaine AD labo.tssr
OS Windows Server 2019 + Windows 10 client
A — Analyse de la demande et contexte technique

Mise en œuvre d'une gestion structurée des accès aux ressources partagées dans un environnement Active Directory. L'objectif est de mettre en place une séparation stricte des accès entre différents services, tout en permettant à la direction d'accéder à des dossiers d'autres services (accès transverse).

Les tâches réalisées :

  • Analyser une demande fonctionnelle simulée portant sur l'accès aux dossiers partagés par service
  • Créer une arborescence de dossiers sur le serveur de fichiers (PARTAGE_DATA)
  • Concevoir et mettre en place des groupes Active Directory selon la méthode AGDLP
  • Configurer les permissions NTFS conformément aux besoins définis
  • Vérifier les accès depuis un poste client avec un compte utilisateur de test
B — Mise en place du partage et des permissions

Création du dossier PARTAGE_DATA sur le serveur de fichiers. Les permissions de partage ont été volontairement simplifiées (Tout le monde : Lecture/Modification), la gestion fine des droits étant assurée par les permissions NTFS, conformément aux bonnes pratiques.

Dans l'OU groupe du domaine labo.tssr, création des groupes globaux : g_direction, g_informatique, g_rh, g_support.

C — Structuration AGDLP
La méthode AGDLP dissocie les comptes utilisateurs des permissions sur les ressources :

Trois groupes locaux de domaine créés pour le dossier RH avec trois niveaux de droits :

Groupe DLDroits NTFSDescription

dl_dossier_RH_CModification (C)Lecture + écriture
dl_dossier_RH_FContrôle total (F)Tous droits
D — Attribution des permissions NTFS

Désactivation de l'héritage des permissions sur le dossier RH. Le groupe Utilisateurs supprimé — les droits sont attribués exclusivement aux groupes locaux de domaine (principe du moindre privilège).

Commandes de référence (PowerShell) :

# Désactiver l'héritage et convertir en règles explicites icacls "E:\PARTAGE_DATA\RH" /inheritance:d # Supprimer le groupe Utilisateurs icacls "E:\PARTAGE_DATA\RH" /remove:g "BUILTIN\Users" # Appliquer les droits par groupe DL icacls "E:\PARTAGE_DATA\RH" /grant "LABO\dl_dossier_RH_R:(OI)(CI)R" icacls "E:\PARTAGE_DATA\RH" /grant "LABO\dl_dossier_RH_C:(OI)(CI)M" icacls "E:\PARTAGE_DATA\RH" /grant "LABO\dl_dossier_RH_F:(OI)(CI)F"
E — Accès transverses : groupe Direction

Le groupe global g_direction est ajouté dans le groupe local de domaine dl_dossier_RH_R. Cela permet à tout membre du service Direction d'accéder en lecture au dossier RH, sans modifier les permissions directes sur le dossier.

F — Validation depuis le poste client
Connexion sur Windows 10 avec un compte membre du groupe g_direction, accès via le chemin réseau \\192.168.240.202\partage_data :
✓ SUCCÈS Dossier RH — accessible en lecture (droits via g_direction → dl_dossier_RH_R)
✗ REFUS Dossier Support — accès refusé (visible mais non autorisé)
La visibilité d'un dossier partagé ne signifie pas un droit d'accès. Cela valide la bonne application des permissions NTFS et le respect du principe du moindre privilège. Les droits sont déterminés uniquement par l'appartenance aux groupes Active Directory.
Moyens utilisés
  • Windows Server 2019 en contrôleur de domaine AD DS + serveur de fichiers
  • Poste client Windows 10 intégré au domaine
  • Console GPMC et Utilisateurs & Ordinateurs Active Directory
  • Package MSI Firefox pour déploiement GPO
03 CCP1 — Ex.3 : Gestion utilisateurs, droits & partages Linux (GLPI)
Environnement technique

Adresse IP 192.168.240.210/24
Application hébergée GLPI (Apache/HTTP)

A — Présentation & B — Enjeux de sécurité

Le serveur héberge GLPI (gestion des incidents/parc informatique) avec plusieurs profils : utilisateurs finaux, techniciens, responsable du support. L'absence de gestion rigoureuse des droits pourrait entraîner des accès non autorisés, des suppressions involontaires et une mauvaise organisation du support.

Démarche appliquée : création de comptes → organisation en groupes → répertoire de partage sécurisé → correction des permissions → tests de validation
C — Création des utilisateurs Linux
Trois profils créés : administrateur système (adminlinux — Charlotte LEROY), technicien informatique (techdupont), utilisateur standard (usermartin).
tssr@GLPI-TP2:~$ sudo adduser adminlinux # Saisie des infos : NOM Charlotte LEROY, B201, 01 45 12 34 10... tssr@GLPI-TP2:~$ sudo adduser techdupont tssr@GLPI-TP2:~$ sudo adduser usermartin # Vérification des comptes créés tssr@GLPI-TP2:~$ id techdupont uid=1002(techdupont) gid=1002(techdupont) groupes=1002(techdupont),100(users) tssr@GLPI-TP2:~$ id usermartin uid=1003(usermartin) gid=1003(usermartin) groupes=1003(usermartin),100(users)
D — Organisation par groupes Linux
# Création des groupes tssr@GLPI-TP2:~$ sudo addgroup techglpi Ajout du groupe « techglpi » (GID 1004)... tssr@GLPI-TP2:~$ sudo addgroup partage Ajout du groupe « partage » (GID 1005)... # Affectation des utilisateurs aux groupes tssr@GLPI-TP2:~$ sudo usermod -aG sudo adminlinux tssr@GLPI-TP2:~$ sudo usermod -aG techglpi techdupont tssr@GLPI-TP2:~$ sudo usermod -aG partage techdupont # Vérification finale tssr@GLPI-TP2:~$ id adminlinux uid=1001(adminlinux) gid=1001 groupes=1001,27(sudo),100(users) tssr@GLPI-TP2:~$ id techdupont uid=1002(techdupont) gid=1002 groupes=1002,100(users),1004(techglpi),1005(partage)
E — Mise en place et sécurisation du partage
# Création du répertoire tssr@GLPI-TP2:~$ sudo mkdir /srv/partage tssr@GLPI-TP2:~$ sudo ls -ld /srv/partage drwxr-xr-x 2 root root 4096 31 janv. 22:02 /srv/partage # Attribution du groupe propriétaire tssr@GLPI-TP2:~$ sudo chown root:partage /srv/partage tssr@GLPI-TP2:~$ sudo ls -ld /srv/partage drwxrwx--- 2 root partage 4096 31 janv. 22:02 /srv/partage # Sécurisation des permissions : 770 → propriétaire+groupe RWX, autres 0 tssr@GLPI-TP2:~$ sudo chmod 770 /srv/partage tssr@GLPI-TP2:~$ sudo ls -ld /srv/partage drwxrwx--- 2 root partage 4096 31 janv. 22:20 /srv/partage
chmod 770 : propriétaire (7=rwx), groupe (7=rwx), autres (0=---). Seuls les membres du groupe partage peuvent accéder au répertoire.

Analyse du site survey : vérification de la cohérence entre l'implantation prévue et le positionnement réel des bornes sur 4 étages. Étude de la répartition des bornes avec code-barres et correspondance sur le fichier de raccordement aux switchs.

Document de raccordement maintenu sous Excel : Access Point, S/N, MAC, Barcode, Interface, Patch Out, Switch, Code, Port, Technical Room.

3 — Analyse de l'architecture réseau

Analyse du chemin de communication borne → switch → contrôleur. Toute rupture dans cette chaîne (switch, VLAN, DHCP, contrôleur) peut bloquer le Wi-Fi.
